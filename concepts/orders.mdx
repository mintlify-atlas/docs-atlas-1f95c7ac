# Orders

Orders are the core data structure in the 1inch Cross-Chain SDK, representing a user's intent to swap tokens across different blockchains.

## Order Structure

All cross-chain orders contain these fundamental components:

### Basic Order Information

```typescript
type OrderInfo = {
  // Source chain asset and amount
  makerAsset: Address
  makingAmount: bigint
  
  // Destination chain asset and amount
  takerAsset: Address
  takingAmount: bigint
  
  // Addresses
  maker: Address        // Who created the order
  receiver?: Address    // Who receives on destination (defaults to maker)
}
```

### Escrow Parameters

These parameters secure the atomic swap:

```typescript
type EscrowParams = {
  hashLock: HashLock           // Cryptographic commitment to secrets
  srcChainId: number           // Source blockchain ID
  dstChainId: number           // Destination blockchain ID
  srcSafetyDeposit: bigint     // Maker's safety deposit
  dstSafetyDeposit: bigint     // Resolver's safety deposit
  timeLocks: TimeLocks         // Time-based execution phases
}
```

### Auction Details

Auctions determine how the exchange rate changes over time:

```typescript
type AuctionDetails = {
  startTime: bigint            // When auction begins (unix timestamp)
  duration: bigint             // How long auction runs (seconds)
  initialRateBump: number      // Initial rate improvement (%)
  points: AuctionPoint[]       // Rate curve points
  gasCost: {
    gasBumpEstimate: bigint
    gasPriceEstimate: bigint
  }
}
```

<Info>
The auction mechanism allows orders to start with favorable rates and gradually become more competitive, incentivizing quick execution while ensuring eventual fulfillment.
</Info>

## Order Lifecycle

Orders progress through several states:

### 1. Created

```typescript
const { hash, quoteId, order } = await sdk.createOrder(quote, {
  walletAddress,
  hashLock,
  preset,
  source,
  secretHashes
})
```

At this stage:
- Order is signed with the maker's private key
- Hash lock commits to secrets without revealing them
- Order is ready for submission but not yet active

### 2. Submitted

```typescript
// EVM orders
const orderInfo = await sdk.submitOrder(
  srcChainId,
  order,
  quoteId,
  secretHashes
)

// Solana orders (requires both off-chain and on-chain submission)
const orderHash = await sdk.announceOrder(order, quoteId, secretHashes)
// Then publish on-chain transaction
```

The order is now:
- Visible to resolvers in the relayer network
- Waiting for a resolver to pick it up
- In the auction phase

### 3. Executing

When a resolver accepts the order:
- Source escrow deployed (locks maker's tokens)
- Destination escrow deployed (locks resolver's tokens)
- Waiting for secret revelation

```typescript
// Monitor for ready escrows
const secretsToShare = await sdk.getReadyToAcceptSecretFills(hash)

for (const { idx } of secretsToShare.fills) {
  // Verify escrow before revealing
  await sdk.submitSecret(hash, secrets[idx])
}
```

### 4. Completed States

**Executed**: Secrets revealed, funds exchanged
```typescript
const { status } = await sdk.getOrderStatus(hash)
// status === OrderStatus.Executed
```

**Expired**: Time locks expired without completion
```typescript
// status === OrderStatus.Expired
```

**Refunded**: Parties reclaimed their original funds
```typescript
// status === OrderStatus.Refunded
```

## EVM Orders

EVM orders use the 1inch Limit Order Protocol with custom escrow extensions.

### EVM Order Type

### Publishing Solana Orders

Solana orders must be published both off-chain and on-chain:

```typescript
import { SvmSrcEscrowFactory } from '@1inch/cross-chain-sdk'
import { web3 } from '@coral-xyz/anchor'

// 1. Announce to relayer (off-chain)
const orderHash = await sdk.announceOrder(order, quoteId, secretHashes)

// 2. Create on-chain instruction
const ix = SvmSrcEscrowFactory.DEFAULT.createOrder(order, {
  srcTokenProgramId: SolanaAddress.TOKEN_PROGRAM_ID
})

// 3. Build and send transaction
const tx = new web3.Transaction().add({
  data: ix.data,
  programId: new web3.PublicKey(ix.programId.toBuffer()),
  keys: ix.accounts.map(a => ({
    isSigner: a.isSigner,
    isWritable: a.isWritable,
    pubkey: new web3.PublicKey(a.pubkey.toBuffer())
  }))
})

const signature = await connection.sendTransaction(tx, [signer])
```

<Warning>
Solana orders are only active after the on-chain transaction confirms. Make sure to wait for confirmation before expecting resolver activity.
</Warning>

## Order Configuration Options

### Multiple Fills

Allow the order to be filled by multiple resolvers in separate transactions:

```typescript
// EVM
const order = EvmCrossChainOrder.new(
  /* ... */,
  { allowMultipleFills: true }
)

// Solana
const order = SvmCrossChainOrder.new(
  /* ... */,
  { allowMultipleFills: true }
)
```

When enabled:
- Order can be partially filled by different resolvers
- Each fill uses a different secret from the merkle tree
- Hash lock must use `forMultipleFills()` with sufficient secrets

### Partial Fills (EVM only)

Allow a single resolver to fill less than the full amount:

```typescript
const order = EvmCrossChainOrder.new(
  /* ... */,
  { allowPartialFills: true }
)
```

### Resolver Whitelist (EVM only)

Restrict who can fill the order during specific time periods:

```typescript
const order = EvmCrossChainOrder.new(
  /* ... */,
  {
    auction: auctionDetails,
    whitelist: [
      {
        address: EvmAddress.fromString('0x...'),
        allowFrom: 0n  // Can fill immediately
      },
      {
        address: EvmAddress.fromString('0x...'),
        allowFrom: 60n // Can fill after 60 seconds
      }
    ]
  }
)
```

### Permits (EVM only)

Include permit signatures to avoid separate approval transactions:

```typescript
// Standard ERC20 permit
const order = EvmCrossChainOrder.new(
  /* ... */,
  { permit: permitSignature }
)

// Permit2
const order = EvmCrossChainOrder.new(
  /* ... */,
  { 
    permit: permit2Signature,
    enablePermit2: true 
  }
)
```

## Common Order Operations

### Getting Multiple Fill Index

Determine which secret index to use for a specific fill amount:

```typescript
const fillIdx = order.getMultipleFillIdx(
  fillAmount,
  remainingAmount
)

const secret = secrets[fillIdx]
const secretHash = HashLock.hashSecret(secret)
const leaves = HashLock.getMerkleLeaves(secrets)
const proof = HashLock.getProof(leaves, fillIdx)
```

From `src/cross-chain-order/base-order.ts:112`:

```typescript
public getMultipleFillIdx(
    fillAmount: bigint,
    remainingAmount = this.makingAmount
): number {
    assert(this.multipleFillsAllowed, 'Multiple fills disabled for order')
    const partsCount = this.hashLock.getPartsCount()

    const calculatedIndex =
        ((this.makingAmount - remainingAmount + fillAmount - 1n) *
            partsCount) /
        this.makingAmount

    if (remainingAmount === fillAmount) {
        return Number(calculatedIndex + 1n)
    }

    return Number(calculatedIndex)
}
```

### Handling Secret Submission

```typescript
const alreadyShared = new Set<number>()

while (orderActive) {
  const ready = await sdk.getReadyToAcceptSecretFills(orderHash)
  
  for (const { idx, srcEscrow, dstEscrow } of ready.fills) {
    if (alreadyShared.has(idx)) continue
    
    // Verify escrow addresses and amounts
    const isValid = await verifyEscrows(srcEscrow, dstEscrow)
    
    if (isValid) {
      await sdk.submitSecret(orderHash, secrets[idx])
      alreadyShared.add(idx)
      console.log(`Shared secret ${idx}`)
    }
  }
  
  await sleep(5000)
}
```

## Best Practices

<Accordion title="Always verify escrows before revealing secrets">
  Before submitting secrets, verify:
  - Escrow addresses match expected values
  - Token addresses are correct
  - Amounts match your order parameters
  - Time locks provide sufficient time
  
  Once a secret is revealed on-chain, anyone can use it.
</Accordion>

<Accordion title="Use recommended presets for production">
  The SDK provides battle-tested presets (fast, medium, slow) that balance:
  - Execution speed
  - Number of secrets required
  - Time lock durations
  - Safety deposit amounts
  
  Only use custom configurations if you understand the tradeoffs.
</Accordion>

<Accordion title="Handle multiple fills correctly">
  When `allowMultipleFills` is enabled:
  - Generate enough secrets (check `preset.secretsCount`)
  - Use merkle tree hash lock (`forMultipleFills`)
  - Track which secrets have been shared
  - Use `getMultipleFillIdx()` to determine the correct secret
</Accordion>

<Accordion title="Monitor order status actively">
  Don't rely on assumptions about order completion:
  - Poll `getOrderStatus()` regularly
  - Check `getReadyToAcceptSecretFills()` for escrow deployments
  - Handle all terminal states (Executed, Expired, Refunded)
  - Implement retry logic for secret submission
</Accordion>
